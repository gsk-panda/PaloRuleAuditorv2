Identifying ZIA vs ZPA issues

ZIA inspects and controls internet-bound traffic. ZPA provides access to internal apps. Use this table to classify the issue quickly.
SymptomLikely serviceFirst check
Can't reach internet sites (Google, SaaS tools)ZIACheck ZIA policy / URL category block
Can't reach internal app (intranet, dev server)ZPACheck ZPA app segment & connector status
SSL error / certificate warning on HTTPS siteZIACheck SSL inspection policy & root CA deployment
App works from office, not remoteZPAZPA tunnel not established; check ZCC login
Slow streaming / video callsZIABandwidth control policy or ZDX media path
Slow internal file shares or DB connectionsZPAApp connector health + latency (ZDX)
File download blocked / quarantinedZIASandbox verdict — check file hash in portal
Specific SaaS tenant blocked (correct site, wrong org)ZIATenant restrictions policy in ZIA
Auth loop / redirect on internal SSOZPAZPA SAML/SCIM config; IdP connector
Partial site load (images load, API calls fail)ZIAURL categories for CDN/API domains; SSL inspection bypass
ZIA

Quick diagnostic steps

  • 1Ask: "What URL or service can't you reach?"
  • 2Have user visit ip.zscaler.com — confirms ZIA tunnel is active
  • 3ZIA portal → Insights → Web Insights for the blocked transaction
  • 4Note block reason: Policy, SSL error, Sandbox, or Firewall
  • 5Verify user's ZIA policy group/profile is correct
  • 6Check if URL is in a blocked category or custom block list
  • 7Test with ZIA bypass / exclude from SSL inspection if cert issue
ZPA

Quick diagnostic steps

  • 1Ask: "What internal app or hostname?" Get exact URL/IP
  • 2Confirm ZPA tunnel state in ZCC tray icon (should show connected)
  • 3ZPA portal → Administration → App Connectors — check health
  • 4Verify app segment covers the destination IP/FQDN and port
  • 5Check user entitlement to the access policy for that segment
  • 6Review ZPA logs for access denied vs connector unreachable
  • 7Test connectivity from a second connector in the same group

ZDX — diagnosing speed and connectivity issues

ZDX provides end-to-end path visibility. Always check ZDX before assuming Zscaler is the problem — the issue may be ISP, cloud provider, or endpoint-side.
ZDX

When to invoke ZDX

  • 1User reports slowness but site is reachable
  • 2Video/VoIP quality issues during calls
  • 3Intermittent timeouts on a specific application
  • 4Issue affects multiple users at one location
  • 5User says "it was faster before Zscaler"
ZDX

Initial data to collect

  • 1Username, device name, OS version
  • 2Affected app name and URL/IP
  • 3Time the issue started (exact or approx.)
  • 4Location — office, home, hotel/public WiFi?
  • 5Does issue persist on mobile hotspot?

ZDX diagnostic workflow

Step 1

Find the user in ZDX

  • ANavigate to ZDX → Users → search by email or username
  • BSelect the affected device and set time range to issue window
  • CReview ZDX Score — below 70 indicates degradation
  • DNote if score drop aligns with user's reported time
Step 2

Path analysis

  • AOpen the affected app's path trace in ZDX
  • BIdentify which segment has high latency or packet loss: Device→ZEN, ZEN→App, App response
  • CIf Device→ZEN high: local ISP or endpoint issue
  • DIf ZEN→App high: app hosting / cloud provider issue
  • ECheck if user is hitting optimal ZEN (Zscaler Enforcement Node)
Step 3

Device health check

  • AZDX → Device → check CPU, RAM, disk at time of issue
  • BCheck WiFi signal strength and channel congestion
  • CVerify ZCC version is current
  • DCheck for dual-tunnel conflict (VPN + ZCC simultaneously)
Step 4

Web probe analysis

  • AReview ZDX web probes for the affected application
  • BCheck DNS resolution time, TCP connect, TLS handshake, TTFB
  • CHigh DNS time → check DNS forwarding profile in ZIA
  • DHigh TLS time → SSL inspection overhead; consider bypass for app
  • EHigh TTFB → origin server or CDN performance issue

Sandbox issues

The Zscaler Sandbox analyzes unknown or suspicious files. Users may be blocked downloading a file even from a trusted domain. This is often misidentified as a connectivity issue.
Sandbox

File download blocked

  • 1Ask: "What is the filename and where were you downloading it from?"
  • 2ZIA portal → Sandbox → Reports — search by filename or hash
  • 3Check verdict: Malicious, Suspicious, Benign, or Unknown/Pending
  • 4If Malicious: do not bypass — escalate to security team
  • 5If Suspicious: review sandbox detonation report; escalate for approval
  • 6If false positive (Benign confirmed): submit hash to allowlist
Sandbox

Pending / hold state

  • 1File is being analyzed — normal for first-seen files (1–3 min typically)
  • 2Ask user to wait and retry the download after a few minutes
  • 3If still pending: check sandbox quota and cloud connectivity
  • 4If urgent and source is trusted: sandbox bypass can be applied per policy by admin
  • 5Log the file hash for tracking; notify security team
Sandbox

Bypass cautions

  • 1Never bypass for Malicious or Suspicious verdicts without security approval
  • 2Bypass should be scoped: specific hash, not whole domain
  • 3Document every bypass request with justification and approver
  • 4Review bypass list quarterly — remove stale entries

URL filtering — detailed troubleshooting

URL filtering is the most common ZIA support category. Always begin with Web Insights to identify the exact transaction before touching any policy.

Step-by-step diagnostic workflow

Step 1

Confirm the block in Web Insights

  • 1ZIA portal → Analytics → Web Insights
  • 2Filter by username and set time range to when issue occurred
  • 3Locate the specific blocked transaction — note the Action column (Block, Allow, Caution)
  • 4Record: URL, category, policy name, rule name, and action applied
  • 5Confirm the transaction matches what the user reported (correct URL, correct time)
Step 2

Check URL categorization

  • 1ZIA portal → Administration → URL Categories → URL Lookup tool
  • 2Enter the full URL (not just domain) — subpaths can have different categories
  • 3Note both the Zscaler category and any custom category overrides
  • 4If the category is wrong: submit recategorization via the lookup tool or help.zscaler.com
  • 5Check if the URL is in any custom allow/block list — custom lists override Zscaler categories
  • 6Watch for wildcard entries (e.g., *.example.com) that may be catching unintended URLs
Step 3

Identify the policy rule

  • 1ZIA portal → Policy → URL & Cloud App Control
  • 2Find the rule name shown in Web Insights — confirm its configuration
  • 3Check which user groups or departments are in scope for that rule
  • 4Verify the user is in the correct group — miscategorized AD/LDAP groups are a common cause
  • 5Check rule order — higher rules take precedence; an early block rule may be catching the user before a later allow rule
  • 6Check time-based scheduling on the rule — may only apply during certain hours
Step 4

Apply an exception (Tier 2)

  • 1Determine the correct scope: user-specific, group-level, or org-wide
  • 2Add the URL to a custom allow list scoped to the appropriate group
  • 3Alternatively: create a new rule above the blocking rule for the specific user/group
  • 4For cloud app control blocks (e.g., personal Dropbox): use tenant restrictions rather than full URL bypass
  • 5Test after applying: changes propagate within 5 minutes typically
  • 6Document the exception with business justification and review date

Advanced checks for persistent URL issues

Advanced

Custom URL category conflicts

  • 1ZIA → Administration → URL Categories — list all custom categories
  • 2Search for the domain across all custom categories — it may appear in multiple
  • 3Custom categories in a block policy always override Zscaler built-in categories
  • 4Check for overly broad wildcards: *.co can inadvertently block country-code TLDs
  • 5Compare custom category rule order — a custom block higher in the policy wins over a custom allow lower down
Advanced

  • 2Cloud app control can block specific functions (e.g., allow Google Drive access but block uploads)
  • 3Web Insights will show "Cloud App Control" as the block reason — distinct from URL policy block
  • 4Check the app's risk score in ZIA — high-risk apps may be auto-blocked by risk-based policy
  • 5For unsanctioned app blocks: confirm with security team before adding exceptions

  • 1ZIA → Policy → URL & Cloud App Control → check for Safe Search enforcement rules
  • 2Safe Search enforcement modifies search result URLs — some apps may break if they detect URL modification
  • 3YouTube restricted mode: check if enforced for the user's group; may block educational content
  • 4Social media "read-only" controls: users can view but not post — common source of "site doesn't work" reports
  • 5Check Web Insights action column — "Allow (Read-Only)" indicates social media restriction is active

ZIA firewall — detailed troubleshooting

ZIA Cloud Firewall controls non-web traffic and enforces rules on ports, protocols, and IP addresses. Firewall blocks often appear as connection timeouts rather than block pages — users may not realise Zscaler is involved.

Step-by-step diagnostic workflow

Step 1

Check ZIA Firewall logs

  • 1ZIA portal → Analytics → Firewall Insights (or Firewall Logs)
  • 2Filter by username + time window of the issue
  • 3Look for Block entries matching the destination IP, port, or protocol
  • 4Record the firewall rule name that triggered the block
  • 5Note: if no firewall log entry exists, the traffic may not be reaching ZIA at all — check ZCC tunnel state
Step 2

Identify the protocol and port

  • 1Ask user: what application is failing? Get the app name and vendor if possible
  • 2Identify the destination port(s) and protocol (TCP/UDP) from Firewall logs
  • 3Common non-standard ports blocked by default: 8080, 8443, 9000, 1433 (SQL), 5432 (Postgres), custom VoIP ports
  • 4Check if UDP traffic is involved — ZIA firewall handles UDP separately from TCP; DNS (53) and VoIP (RTP) are common UDP issues
  • 5For ICMP (ping) failures: ZIA blocks ICMP by default unless a rule explicitly permits it
Step 3

Review the firewall policy

  • 1ZIA portal → Policy → Firewall Control
  • 2Locate the rule name from Firewall Insights — confirm it is intentionally blocking the traffic
  • 3Check rule order — rules are evaluated top-to-bottom; a broad "Block All" rule near the top may override a lower allow rule
  • 4Verify the user's network location (IP/subnet) and whether they fall into the rule's source scope
  • 5Check if the rule is scoped to a specific location (branch) vs all locations
  • 6Look for implicit deny at the bottom of the ruleset — all unmatched traffic is dropped
Step 4

Create or modify a firewall rule (Tier 2)

  • 1Confirm with security team that the traffic is legitimate before creating an allow rule
  • 2Create a new rule above the blocking rule — scope it as narrowly as possible (specific user group, destination IP/FQDN, port)
  • 3Use Application objects where possible (ZIA has built-in app definitions for common tools) rather than raw port rules
  • 4Avoid creating broad "Allow Any/Any" rules — scope to the minimum required access
  • 5Test and verify in Firewall Insights after applying — changes propagate within ~5 minutes
  • 6Set a review date on the rule — document the business justification

Advanced firewall checks

Advanced

DNS control vs URL filtering

  • 1DNS Control blocks occur before the HTTP request is made — user gets NXDOMAIN, not a block page
  • 2ZIA → Policy → DNS Control — check for domain entries matching the failed destination
  • 3DNS blocks are also used for threat intelligence feeds — domain may be flagged as C2 or phishing
  • 4Test: nslookup domain.com from the affected device — NXDOMAIN confirms DNS-level block
  • 5Check if browser DoH (DNS-over-HTTPS) is bypassing ZIA DNS — disable in browser settings
Advanced

Bandwidth control & QoS

  • 1ZIA → Policy → Bandwidth Control — check if a rule is throttling the user's traffic type
  • 2Bandwidth control does not block — it rate-limits; symptoms are slow speeds, not timeouts
  • 3Check which category is being throttled and confirm the user's traffic matches that category
  • 4Priority apps (Teams, Zoom, Office 365) should be in high-priority QoS classes — verify in policy
  • 5Escalate to Tier 2 to adjust bandwidth policy if legitimate business traffic is being throttled
Advanced

Branch / location firewall isolation

  • 1ZIA → Administration → Locations — confirm the branch is configured with the correct subnet/IP range
  • 2If a branch IP is not in any location, traffic may be processed under the "Other" default location with different policies
  • 3Check GRE/IPSec tunnel health for the branch — if the tunnel is down, traffic may route directly without ZIA inspection
  • 4Confirm that location-specific firewall rules are applied in the correct location group
  • 5Escalate to Tier 2/3 if branch tunnel is unstable — involves ISP and Zscaler ZEN coordination

SSL decryption — detailed troubleshooting

SSL inspection is the most complex ZIA component and the most common source of application breakage. Zscaler acts as a man-in-the-middle — it terminates the user's TLS session, inspects the content, and re-encrypts it. Any app that validates the certificate chain strictly may fail.

How SSL inspection works — key concepts

Concept

The ZIA root CA

When ZIA inspects HTTPS traffic, it re-signs the server certificate using the Zscaler Root CA. The device must trust this CA — if it does not, the browser shows a certificate error. The root CA must be deployed to every managed device via MDM/GPO before SSL inspection is enabled.

Concept

Certificate pinning

Some applications (mobile apps, banking clients, security tools, OS update services) use certificate pinning — they only accept a specific certificate or CA, not Zscaler's re-signed cert. These apps will always fail under SSL inspection and must be added to the SSL bypass list.

Concept

What bypasses SSL inspection

Bypassed traffic is still forwarded through ZIA but is not inspected — Zscaler passes the original server certificate through to the client. Bypasses can be scoped by URL category, specific URL, source user/group, destination IP, or application type.

Step-by-step SSL diagnostic workflow

Step 1

Confirm SSL inspection is the cause

  • 1Ask user to click the padlock icon in their browser on the failing site
  • 2If the issuer shows "Zscaler" — SSL inspection is active on this site
  • 3If the issuer shows the real server cert — SSL inspection is already bypassed; the issue is elsewhere
  • 4ZIA → Analytics → Web Insights → find the transaction → check the SSL Inspected column
  • 5For non-browser apps: check if the app logs show "certificate verification failed" or "untrusted CA"
Step 2

Check root CA deployment

  • 1ZCC → About → Certificates — confirm Zscaler root CA is listed as Trusted
  • 2Windows: certmgr.msc → Trusted Root Certification Authorities → verify Zscaler CA is present
  • 3macOS: Keychain Access → System → Certificates → search "Zscaler" → confirm Trust is set to "Always Trust"
  • 4Firefox uses its own certificate store — the root CA must be pushed to Firefox separately via GPO or Firefox policy
  • 5If root CA is missing: escalate to Tier 2 for MDM/GPO re-deployment of the CA cert
Step 3

Identify if site needs a bypass

  • 1Test: does the app/site work if temporarily ZCC is disabled? If yes, SSL inspection is the likely cause
  • 2Look up the domain in ZIA → Policy → SSL Inspection — check if it is already in the bypass list
  • 3Check built-in ZIA bypass categories (Financial, Health, Government) — these are typically pre-bypassed
  • 4Research the app: does the vendor documentation mention certificate pinning? Common examples: Apple services, Windows Update, Okta, Duo, banking apps, many mobile apps
  • 5Check ZIA's built-in "Do Not Decrypt" list — Zscaler maintains a pre-configured list of known pinned-cert services
Step 4

Add SSL bypass (Tier 2)

  • 1ZIA → Policy → SSL Inspection → Do Not Decrypt rules
  • 2Add a "Do Not Decrypt" rule for the specific URL, domain, or URL category
  • 3Scope the bypass as narrowly as possible — by user group and/or destination domain
  • 4Avoid bypassing entire URL categories unless necessary (e.g., do not bypass all "Financial Services" — bypass only the specific domain)
  • 5Verify the bypass is working: browser padlock should now show the real server certificate, not Zscaler's
  • 6Document the bypass with reason and review date — SSL bypasses reduce inspection coverage

SSL inspection escalation criteria

Escalate to Tier 3

When to involve Zscaler TAC

  • !Persistent SSL inspection failures affecting multiple users after CA deployment verified
  • !ZIA re-signing with incorrect intermediate chain causing widespread cert errors
  • !ECH (Encrypted Client Hello) or TLS 1.3 edge cases that bypass inspection unexpectedly
  • !HPKP (HTTP Public Key Pinning) violations causing browser trust failures
  • !Zscaler CA certificate itself approaching expiry (check ZIA → Administration → Certificates)
Best practice

SSL bypass governance

  • 1Maintain a bypass register — every Do Not Decrypt entry should have: owner, date added, reason, review date
  • 2Review bypass list quarterly — remove entries for apps that have been decommissioned or updated
  • 3Use ZIA's pre-built "Do Not Decrypt" categories (Financial, Health, Government) rather than manual entries where possible
  • 4Never bypass entire broad categories like "Business and Economy" — always scope to the minimum required domain
  • 5After adding a bypass, verify in Web Insights that the site now shows "Not Inspected" — confirm the bypass applied

Escalation framework

Tier 1 — Service desk

Handle: ZCC restart, login help, log collection, URL category lookups, confirming ZIA/ZPA tunnel state, relaying block reasons, ZDX score check. Escalate after 15 min if unresolved.

Tier 2 — Network/security admin

Handle: Policy changes (URL allow/block, SSL bypass), ZPA app segment creation/editing, connector health issues, sandbox verdict disputes, ZCC enrollment fixes, ZDX path analysis.

Tier 3 — Zscaler TAC / architect

Handle: ZEN-level issues, cloud-side outages, complex policy conflicts, ZPA connector cluster failures, persistent ZDX anomalies, sandbox false positive submissions, IdP/MDM integration issues.
